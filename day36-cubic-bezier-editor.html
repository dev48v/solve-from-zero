〰️ Cubic-Bézier Easing Editor — the four numbers are just two control points on a time→progress curve

Drag the two dots, watch the motion, copy the exact cubic-bezier(…).

A CSS easing is not a shape you draw — it is a timing curve: the horizontal axis is time (0 → 1) and the vertical axis is how far the animation has progressed (0 → 1). The curve always starts at the bottom-left (0,0) and ends at the top-right (1,1); the only things you control are the two handles that bend it, and those handles are exactly the four numbers cubic-bezier(x1, y1, x2, y2). Pull a handle up early and motion starts fast; pull it flat and motion crawls then snaps. Push a handle past the top or below the bottom and progress overshoots 100% or dips below 0 — that is how a "bounce" or an "anticipation" is born. Drag below, hit a preset, and read the value you paste straight into CSS. No library, all vanilla JS + canvas.

Five things that surprise people

        
1. The axes are time and progress — not X and Y position. The curve does not describe a path across the screen; it maps "how much time has passed" to "how far along the animation is". A steep part = fast motion, a flat part = the element barely moving. 2. t is not x. A Bézier is drawn by a hidden parameter t, but the browser needs the y at a given time x — and x(t) is itself a curve, so you must solve for the t whose x equals your time before you can read y. That solve is the only real math in the whole tool. 3. x is clamped, y is not. The two control-point x values are forced into 0–1 because time can't reverse; the y values are free — and y going above 1 or below 0 is precisely what makes progress overshoot (bounce) or back up first (anticipation). 4. The named keywords are just presets. ease is literally cubic-bezier(.25,.1,.25,1), ease-in-out is (.42,0,.58,1), and linear is the straight diagonal (0,0,1,1) — nothing magic, just four numbers. 5. Ease-out feels best for things arriving. Fast-then-slow (ease-out) reads as an object settling into place; slow-then-fast (ease-in) reads as an object leaving. Using the wrong one is why some UIs feel sluggish or abrupt.

🔨 Build it yourself

    
A cubic-bézier editor has exactly one hard idea and a lot of easy plumbing. The hard idea: the curve is drawn by a hidden parameter t, but an easing must answer "at time x, how far along am I?" — and because the x-axis is also a curve of t, you have to solve for the t whose x equals your time, then read y there. Everything else is plumbing: store four numbers, map the unit square to canvas pixels, draw a sampled curve, drag two dots, and run one animation clock. There is no library — the entire engine is the handful of functions below, and it is the same math browsers run internally.

    
    
      
  1. 
        
    1
    The whole state is four numbers
    
        
    A cubic Bézier easing has four fixed points: it always begins at (0,0) and ends at (1,1), so those never move. The only thing you edit is the two control points P1=(x1,y1) and P2=(x2,y2) — which is why the CSS function takes exactly four arguments. Keep them in one plain object; that object is your document.
    
        
    // cubic-bezier(x1, y1, x2, y2)
// endpoints are fixed: P0 = (0,0)  and  P3 = (1,1)
// you only ever move the two control points:
let curve = { x1: 0.25, y1: 0.1, x2: 0.25, y2: 1.0 };  // this is "ease"
// x1, x2 will be clamped to 0..1 (time); y1, y2 are free (can overshoot)
    
      
    2. 
      
  2. 
        
    2
    Turn one axis into a cubic polynomial
    
        
    Each axis of the curve (x and y separately) is a cubic in t. With the endpoints fixed at 0 and 1, the Bézier collapses to the tidy polynomial a·t³ + b·t² + c·t. Pre-computing the three coefficients from the two control values (this is the classic WebKit "UnitBezier" form) makes evaluating the curve — and later its slope — a couple of multiplications.
    
        
    // one axis, control values a1 & a2  (endpoints are 0 and 1)
function coeffs(a1, a2){
  const c = 3 * a1;
  const b = 3 * (a2 - a1) - c;
  const a = 1 - c - b;          // forces the axis to reach 1 at t = 1
  return { a, b, c };
}
// value of that axis at parameter t:  a t³ + b t² + c t
function bezier(t, a1, a2){
  const { a, b, c } = coeffs(a1, a2);
  return ((a * t + b) * t + c) * t;   // Horner form
}
    
      
    3. 
      
  3. 
        
    3
    Its derivative (for Newton's method)
    
        
    To solve the curve quickly we need its slope, dx/dt. Differentiating a·t³ + b·t² + c·t gives 3a·t² + 2b·t + c — another two-multiply expression. This is the "which way and how steeply is x changing?" that Newton-Raphson will use to home in on the right t.
    
        
    // d/dt of  a t³ + b t² + c t  =  3a t² + 2b t + c
function bezierD(t, a1, a2){
  const { a, b, c } = coeffs(a1, a2);
  return (3 * a * t + 2 * b) * t + c;
}
    
      
    4. 
      
  4. 
        
    4
    The crux: solve x(t) = time for t
    
        
    This is the only real algorithm. The animation gives you a time (0–1); you need the t whose x-curve equals that time. Newton-Raphson converges in a handful of steps because t ≈ x is already a great starting guess. If the slope goes flat (a nearly vertical curve) Newton can stall, so fall back to plain bisection, which always brackets the answer. This is exactly what browser engines do.
    
        
    function solveT(x, x1, x2){
  if (x <= 0) return 0;
  if (x >= 1) return 1;
  let t = x;                                 // x is a great first guess
  for (let i = 0; i < 8; i++){               // Newton-Raphson
    const err = bezier(t, x1, x2) - x;
    if (Math.abs(err) < 1e-6) return t;
    const d = bezierD(t, x1, x2);
    if (Math.abs(d) < 1e-6) break;           // flat slope -> bisect instead
    t -= err / d;
  }
  let lo = 0, hi = 1; t = x;                 // bisection fallback (always works)
  for (let i = 0; i < 40; i++){
    const xv = bezier(t, x1, x2);
    if (Math.abs(xv - x) < 1e-6) break;
    if (xv < x) lo = t; else hi = t;
    t = (lo + hi) / 2;
  }
  return t;
}
    
      
    5. 
      
  5. 
        
    5
    The easing function itself
    
        
    With the solver in hand the easing is one line: find the t for the given time on the x axis, then read the y axis at that same t. Time in, eased progress out. Every property you animate — position, scale, opacity — multiplies its range by this returned value.
    
        
    function ease(x, x1, y1, x2, y2){
  const t = solveT(x, x1, x2);   // find t where the x-curve == time
  return bezier(t, y1, y2);      // read the y-curve at that t
}
// ease(0.5, 0.42, 0, 0.58, 1)  ->  0.5   (ease-in-out is symmetric at the middle)
    
      
    6. 
      
  6. 
        
    6
    Map the unit square to canvas pixels
    
        
    The maths lives in a 0–1 world; the screen is pixels with y pointing down. Two helpers convert between them. Leave vertical head-room (map a range like -0.6 … 1.6) so overshooting curves don't clip off the top or bottom, and remember to flip y so progress grows upward like a graph.
    
        
    const PADX = 30, MT = 18, MB = 18;   // pixel margins
const YMIN = -0.6, YMAX = 1.6;       // value range shown (head-room for overshoot)
const toPx = x => PADX + x * (W - 2*PADX);
const toPy = y => H - MB - ((y - YMIN)/(YMAX - YMIN)) * (H - MT - MB);  // y flipped
const fromPx = px => (px - PADX) / (W - 2*PADX);
const fromPy = py => YMIN + ((H - MB) - py)/(H - MT - MB) * (YMAX - YMIN);
    
      
    7. 
      
  7. 
        
    7
    Draw the curve by sampling t
    
        
    To draw the curve you don't need the solver at all — just walk t from 0 to 1 in small steps and plot the point (x(t), y(t)). Draw the faint diagonal (that's linear) for reference, the two handle lines from each endpoint to its control point, then the curve on top. The handle lines are the visual cue that these dots "pull" the curve toward themselves.
    
        
    function drawCurve(c){
  ctx.beginPath();
  for (let i = 0; i <= 100; i++){
    const t = i / 100;
    const x = bezier(t, c.x1, c.x2);
    const y = bezier(t, c.y1, c.y2);
    i ? ctx.lineTo(toPx(x), toPy(y)) : ctx.moveTo(toPx(x), toPy(y));
  }
  ctx.strokeStyle = "#f59e0b"; ctx.lineWidth = 3; ctx.stroke();
  handleLine(0, 0, c.x1, c.y1);   // (0,0) -> P1
  handleLine(1, 1, c.x2, c.y2);   // (1,1) -> P2
  dot(c.x1, c.y1, "#f59e0b");     // draggable P1
  dot(c.x2, c.y2, "#3b82f6");     // draggable P2
}
    
      
    8. 
      
  8. 
        
    8
    Drag the two control points
    
        
    On pointer-down, find which handle (if any) is under the cursor in pixel space. On move, convert the cursor back to curve space and write it into the dragged point — clamping x to 0–1 (time can't reverse) while leaving y free (so you can drag a handle above 1 or below 0 to create overshoot). Pointer events cover mouse and touch with one code path.
    
        
    let dragging = null;               // "P1" | "P2" | null
canvas.addEventListener("pointerdown", e => {
  const [mx, my] = mouse(e);
  if (near(mx, my, curve.x1, curve.y1)) dragging = "P1";
  else if (near(mx, my, curve.x2, curve.y2)) dragging = "P2";
});
canvas.addEventListener("pointermove", e => {
  if (!dragging) return;
  const [mx, my] = mouse(e);
  const x = Math.min(1, Math.max(0, fromPx(mx)));   // clamp time to 0..1
  const y = fromPy(my);                             // progress is free
  if (dragging === "P1"){ curve.x1 = x; curve.y1 = y; }
  else                  { curve.x2 = x; curve.y2 = y; }
  sync();                                           // redraw + refresh CSS output
});
addEventListener("pointerup", () => dragging = null);

  9. 
        
    9
    One clock drives every demo
    
        
    A single requestAnimationFrame loop keeps a clock in milliseconds. The current time is clock / duration (capped at 1), and the eased value is ease(time, …). Feed that one number to everything — the ball's position, a scale, an opacity, a rotation, and a marker on the curve — so they stay perfectly in sync. Add a short hold at the end before wrapping so the loop is readable.
    
        
    let clock = 0, last = 0, playing = true;
const HOLD = 450;                              // pause at the end of each loop
function frame(ts){
  const dt = ts - (last || ts); last = ts;
  if (playing) clock += dt;
  if (clock >= duration + HOLD) clock = 0;      // loop
  const time = Math.min(clock / duration, 1);  // x axis  (0..1)
  const p = ease(time, curve.x1, curve.y1, curve.x2, curve.y2);
  ballEased.style.left  = (12 + p * TRACK) + "px";   // eased
  ballLinear.style.left = (12 + time * TRACK) + "px"; // linear ghost
  pScale.style.transform  = `scale(${0.4 + p * 0.9})`;
  pFade.style.opacity     = Math.max(0, Math.min(1, p));  // opacity clamps!
  pRotate.style.transform = `rotate(${p * 360}deg)`;
  requestAnimationFrame(frame);
}
requestAnimationFrame(frame);

    Serialise to CSS + presets are just data
    
        
    The output is one template string — round the four numbers and wrap them in cubic-bezier(…) — copied to the clipboard in a single call. And the named easings you reach for daily are nothing but canned versions of the same four numbers, so a preset is just an object you drop into state. That's the whole tool.
    
        
    const r = n => Math.round(n * 1000) / 1000;
const toCss = c => `cubic-bezier(${r(c.x1)}, ${r(c.y1)}, ${r(c.x2)}, ${r(c.y2)})`;

copyBtn.onclick = () => navigator.clipboard.writeText(toCss(curve));

const PRESETS = {                       // the CSS keywords, decoded
  "linear":      { x1:0,    y1:0,   x2:1,    y2:1   },
  "ease":        { x1:0.25, y1:0.1, x2:0.25, y2:1   },
  "ease-in":     { x1:0.42, y1:0,   x2:1,    y2:1   },
  "ease-out":    { x1:0,    y1:0,   x2:0.58, y2:1   },
  "ease-in-out": { x1:0.42, y1:0,   x2:0.58, y2:1   },
  "ease-out-back": { x1:0.34, y1:1.56, x2:0.64, y2:1 }  // overshoots past 1 -> springy
};

🎉 Day 36 of SolveFromZero done.
You built a real cubic-bézier easing editor with no library: a four-number state, a WebKit-style polynomial for each axis (a·t³+b·t²+c·t), a solveT() that uses Newton-Raphson (with a bisection safety net) to find the t for a given time, and an ease() that reads y there — the exact algorithm a browser runs. On top sit the easy parts: a canvas that maps the unit square to pixels and draws the curve by sampling t, two draggable control points (x clamped to time, y free for overshoot), and one animation clock feeding position, scale, opacity and rotation at once.
