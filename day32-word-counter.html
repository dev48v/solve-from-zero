The traps a one-liner falls into

1. .length is code units, not characters. A single emoji is a surrogate pair (2), a flag is two regional-indicators (4), a ZWJ family can be 8 вҖ” yet each is one grapheme. Counting storage size over-counts anything past the Basic Multilingual Plane. 2. Even code points aren't enough. cafГ© can be 4 code points (Г© precomposed) or 5 (e + a combining accent) for the same visible word вҖ” only grapheme segmentation gives the reader's count. 3. Words aren't just split(" "). Punctuation, em-dashes, apostrophes, and scripts written without spaces (Chinese, Japanese, Thai) break naГҜve splitting; Intl.Segmenter with granularity:"word" and isWordLike is locale-aware. 4. Sentence & paragraph counts are heuristics. "Mr. Smith" has one period but isn't two sentences вҖ” treat these as estimates, not truth. 5. The limits mean different things. Twitter weights URLs and CJK differently; SMS is 160 only in GSM-7 вҖ” one emoji flips the whole message to UCS-2 and the cap drops to 70.

рҹ”Ё Build it yourself

    
A counter is a pipeline: take the raw string, and derive every statistic from the right unit. The one decision that separates a toy from a correct tool is refusing to use String.length for anything a human calls "a character" вҖ” you segment into graphemes instead. From the grapheme and word lists, sentences, paragraphs, lines, reading time, keyword density and the platform limit meters all fall out. Everything below is vanilla JS with no library вҖ” the exact code powering the LOOK tab. In production you rarely need a package for this; the platform's own Intl.Segmenter is the library.

    Count graphemes, not .length
    
        
    The whole tool hinges on this. Intl.Segmenter with granularity:"grapheme" splits text into user-perceived characters, so an emoji ZWJ sequence or a base-letter-plus-combining-mark counts as one. Keep a spread fallback ([...s]) for ancient engines вҖ” it gets code points right (better than .length) even if it can't merge ZWJ clusters.
    
        
    function graphemes(str){
  if (typeof Intl !== "undefined" && Intl.Segmenter){
    const seg = new Intl.Segmenter(undefined, { granularity: "grapheme" });
    return [...seg.segment(str)].map(s => s.segment);
  }
  return [...str];                 // fallback: code points, not code units
}
const charCount = graphemes(text).length;          // вң“ what a reader sees
// contrast: text.length (code units)  vs  [...text].length (code points)

    Characters with vs without spaces
    
        
    Two standard numbers. "With spaces" is the full grapheme count; "without spaces" filters out every whitespace grapheme (space, tab, newline). Filtering on the grapheme list вҖ” not the raw string вҖ” keeps the count correct when the text also holds emoji.

const withoutSpaces = gr.filter(g => !/^\s+$/.test(g)).length;

    Words the locale-aware way
    
        
    text.split(/\s+/) breaks on the very first hyphenated term or CJK sentence. The right tool is Intl.Segmenter at granularity:"word": it yields segments flagged isWordLike, so punctuation and spaces are dropped and scripts without spaces still segment. A Unicode regex is the fallback.
    
        
    function words(text){
  if (typeof Intl !== "undefined" && Intl.Segmenter){
    const seg = new Intl.Segmenter(undefined, { granularity: "word" });
    return [...seg.segment(text)].filter(s => s.isWordLike).map(s => s.segment);
  }
  return text.match(/[\p{L}\p{N}]+(?:['вҖҷ-][\p{L}\p{N}]+)*/gu) || [];
}
const wordCount = words(text).length;

    Sentences, paragraphs, lines (heuristics)
    
        
    These are estimates, not grammar. A sentence ends at . ! ? вҖҰ вҖ” so split on runs of those and count non-empty pieces. A paragraph is a block separated by a blank line. A line is literally a \n-delimited row. Be honest that "Mr. Smith" fools the sentence rule; that's the accepted trade-off every counter makes.
    
        
    const sentences = text.trim()
  ? text.split(/[.!?вҖҰ]+/).map(s => s.trim()).filter(Boolean).length : 0;
const paragraphs = text.trim()
  ? text.split(/\n\s*\n/).map(s => s.trim()).filter(Boolean).length : 0;
const lines = text === "" ? 0 : text.split(/\r\n|\r|\n/).length;

    Reading & speaking time
    
        
    Both are just words Г· rate. Silent reading averages ~200вҖ“250 words per minute; speaking aloud is slower, ~130 wpm (the number voice-over and presentation guides use). Multiply by 60 for seconds, then format. It's an estimate вҖ” dense or technical text reads slower вҖ” but it's the number every blog "5 min read" badge is built on.
    
        
    const readingSec  = wordCount / 200 * 60;   // ~200 wpm silent reading
const speakingSec = wordCount / 130 * 60;   // ~130 wpm read aloud

function fmt(sec){
  if (sec <= 0) return "0s";
  const m = Math.floor(sec / 60), s = Math.round(sec % 60);
  return m ? m + "m" + (s ? " " + s + "s" : "") : s + "s";
}

    Keyword density + stop-words
    
        
    Lower-case the words, drop stop-words (the, and, of вҖ” function words that dominate every text and carry no topic), tally frequencies, and sort. Density is a word's count divided by the total word count. Writers use it to catch keyword-stuffing; anything spiking above a few percent reads unnaturally.
    
        
    const STOP = new Set("the a an and or but of to in on for with as is are was".split(" "));
function density(text, topN){
  const ws = words(text).map(w => w.toLowerCase());
  const total = ws.length, freq = {};
  for (const w of ws) if (!STOP.has(w) && w.length > 1) freq[w] = (freq[w] || 0) + 1;
  return Object.entries(freq).sort((a,b) => b[1] - a[1]).slice(0, topN)
    .map(([word, count]) => ({ word, count, pct: count / total * 100 }));
}

    The 280 / 160 limit meters
    
        
    A tweet is 280; a meta description renders to about 160; an SMS is 160 вҖ” but only in GSM-7, the 7-bit alphabet feature phones used. The instant a character falls outside GSM-7 (any emoji, curly quote, or non-Latin letter) the whole message switches to 16-bit UCS-2 and the limit collapses to 70. So the SMS meter must first decide the encoding, then pick the cap.
    
        
    function smsInfo(text){
  const gsm = /^[@ВЈ$ВҘГЁГ©Г№Г¬ГІГҮ\nГҳГё\rГ…ГҘО”_ОҰО“ОӣО©О ОЁОЈОҳОһ ГҶГҰГҹГү !"#ВӨ%&'()*+,\-.\/0-9:;<=>?ВЎA-ZГ„Г–Г‘ГңВ§Вҝa-zГӨГ¶ГұГјГ \^{}\\\[~\]|вӮ¬]*$/;
  const isGsm = gsm.test(text);
  return { encoding: isGsm ? "GSM-7" : "UCS-2", limit: isGsm ? 160 : 70 };
}
function meter(count, limit){                 // bar fills, then goes red past cap
  return { pct: Math.min(100, count / limit * 100), over: count > limit };
}
