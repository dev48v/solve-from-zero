📐 Responsive Breakpoint Previewer — paste HTML+CSS, drag a live viewport, watch which @media breakpoints fire

Drag the width, pick a device, and watch the layout reflow as each breakpoint lights up.

A responsive page has no single "size" — it renders differently at every viewport width, and the switches happen at breakpoints: widths where a @media query flips a chunk of CSS on or off. The hard part of building responsive layouts is that you can only see one width at a time in a normal browser, so you resize the whole window and squint. This tool fixes that by rendering your HTML+CSS inside an isolated <iframe> whose width you control independently of the page. Because a media query is evaluated against its own document's viewport, shrinking the iframe to 390px makes the preview behave exactly as it would on an iPhone — @media (min-width:768px) stays off, the mobile layout shows — while the real page around it is still 1400px wide. Drag the handle (or the ruler, or the slider) and you scrub through every width continuously; the panel on the right shows which of the common Tailwind-style breakpoints (sm 640 · md 768 · lg 1024 · xl 1280) are active, and which of your own @media rules are currently firing. No library — just iframe.srcdoc, a width in pixels, and a little arithmetic comparing the width to each breakpoint.

Five things that surprise people

1. The iframe has its own viewport. Media queries inside srcdoc are measured against the iframe's width, not the browser window — that is the whole trick that lets a 390px "phone" live inside a 1400px page. Without the iframe you'd have to resize the actual window. 2. min-width is mobile-first, max-width is desktop-first. @media (min-width:768px) adds styles as you grow past 768; @media (max-width:767px) adds them as you shrink below 768. Mixing the two families in one stylesheet is the classic source of "why are both rules applying?" — because at some widths both can be true. 3. Breakpoints stack. At 1300px every one of sm/md/lg/xl min-width query is simultaneously matched; the one that wins is simply the last in source order (or the most specific), not "the biggest". Order your media queries small-to-large. 4. The famous off-by-one. If you write max-width:768px and min-width:768px, both fire at exactly 768. Pick max-width:767.98px (or 767px) so the ranges don't overlap — this is why frameworks use those odd .98 values. 5. Breakpoints are content, not devices. 640/768/1024/1280 are conventions, not phone sizes; the right place for a breakpoint is wherever your layout starts to look bad. Drag the ruler until it breaks, and put the query there.

🔨 Build it yourself

    
A breakpoint previewer looks fancy but has almost no logic — the browser does all the layout, twice: once for your page and once, independently, inside the iframe. The whole tool is an iframe.srcdoc holding the user's HTML+CSS, a single number (the frame's width in pixels), and three ways to change that number (buttons, a slider, a drag handle). Everything on the right is just arithmetic: compare the current width to each breakpoint value, and to each min/max-width you scraped out of the CSS with a regex. The one real idea to internalise is iframe isolation — a media query is evaluated against its own document's viewport, so an iframe you can resize is a device you can hold in your hand. There is no library.

    
    
      
  1. 
        
    1
    The state is tiny: source strings + one width
    
        
    The source of truth is three values: the HTML the user typed, the CSS they typed, and the current viewport width. The list of breakpoints is fixed data. Everything the UI shows is derived from these.
    
        
    let html  = DEFAULT_HTML;   // markup in the textarea
let css   = DEFAULT_CSS;    // styles in the textarea
let curW  = 768;            // the iframe's width, in px

const BREAKPOINTS = [
  { name:"sm", w:640 }, { name:"md", w:768 },
  { name:"lg", w:1024 }, { name:"xl", w:1280 }
];
const PRESETS = [
  { name:"iPhone", w:390 },  { name:"iPad", w:768 },
  { name:"Laptop", w:1024 }, { name:"Desktop", w:1440 }
];

  2. 
        
    2
    Render into an isolated iframe with srcdoc
    
        
    Wrap the user's markup + CSS into a full mini-document and hand it to the iframe's srcdoc attribute. The iframe now has its own viewport, so its media queries respond to its width — not the page's. That is the isolation trick the whole tool rests on.
    
        
    function buildDoc(html, css){
  return '<!doctype html><html><head><meta charset="utf-8">'
       + '<meta name="viewport" content="width=device-width,initial-scale=1">'
       + '<style>' + css + '</style></head><body>'
       + html + '</body></html>';
}
// srcdoc = a document string; no server, no file, no cross-origin fetch
pv.srcdoc = buildDoc(html, css);

  3. 
        
    3
    Width is just a CSS pixel value on the frame
    
        
    There is no "resize" API. You set the wrapper's style.width in pixels; the iframe fills it at 100%, so changing the number literally changes the viewport the preview sees. Clamp it to sane bounds and re-derive the panels.
    
        
    const MINW = 320, MAXW = 1600;
function setWidth(w){
  curW = Math.round(Math.min(MAXW, Math.max(MINW, w)));
  frame.style.width = curW + "px";   // iframe is width:100% inside
  refresh();                         // update readout + panels
}
// no iframe rebuild here — resizing is instant & smooth

    4
    Three inputs, one setter — presets & slider
    
        
    Every control just calls setWidth. A preset is a canned number, so the buttons are pure data; the slider passes its value through directly. One function is the single funnel that keeps the readout, ruler and panels in sync no matter which control moved.
    
        
    PRESETS.forEach(p => {
  const b = document.createElement("button");
  b.textContent = p.name; b.dataset.w = p.w;
  b.onclick = () => setWidth(p.w);   // a preset is just a width
  presetRow.appendChild(b);
});
slider.oninput = () => setWidth(+slider.value);

  5. 
        
    5
    The drag handle — pointer events + capture
    
        
    The tactile control is a bar on the frame's right edge. On pointerdown record the start X and start width; on pointermove add the delta. setPointerCapture means the drag keeps working even when the cursor leaves the little handle — the difference between a janky and a solid resizer.
    
        
    let startX, startW, dragging = false;
handle.addEventListener("pointerdown", e => {
  dragging = true; startX = e.clientX; startW = curW;
  handle.setPointerCapture(e.pointerId);   // keep events while dragging
});
handle.addEventListener("pointermove", e => {
  if (dragging) setWidth(startW + (e.clientX - startX));
});
handle.addEventListener("pointerup", () => { dragging = false; });

  6. 
        
    6
    Which common breakpoints are active?
    
        
    Mobile-first breakpoints are trivial to evaluate: a breakpoint is active when the width is greater-than-or-equal to its value. The "top" breakpoint — the one whose styles win — is simply the largest active one. This is the entire breakpoint panel.
    
        
    function activeAt(w){
  return BREAKPOINTS.filter(b => w >= b.w);   // min-width semantics
}
function topBreakpoint(w){
  let t = null;
  BREAKPOINTS.forEach(b => { if (w >= b.w) t = b; });
  return t;   // largest active — last to apply in the cascade
}
// at 1024 → active: sm, md, lg   ·   top: lg

  7. 
        
    7
    Scrape the user's @media rules with a regex
    
        
    To show which of their queries fire, read every @media header out of the CSS and pull each min/max-width feature. You don't need a real CSS parser — a regex over the at-rule headers is plenty for width queries, which are 99% of what people write.
    
        
    function parseMedia(css){
  const out = [], head = /@media([^{]+)\{/g; let m;
  while ((m = head.exec(css))) {
    const feats = [], f = /(min|max)-width\s*:\s*(\d+)\s*px/g; let x;
    while ((x = f.exec(m[1]))) feats.push({ kind:x[1], value:+x[2] });
    if (feats.length) out.push({ cond:m[1].trim(), feats });
  }
  return out;
}
    
      
    8. 
      
  8. 
        
    8
    Does a rule fire at this width?
    
        
    A media block fires when all its features are satisfied (they're AND-ed). min-width passes when the width is at least the value; max-width passes when it's at most the value. This one predicate drives the FIRES / idle badge for every parsed rule.
    
        
    function mediaFires(item, w){
  return item.feats.every(f =>
    f.kind === "min" ? w >= f.value    // grow past it
                     : w <= f.value);  // shrink below it
}
// (min-width:640px) and (max-width:1023px) → a RANGE:
// fires only when 640 <= w <= 1023
    
      
    9. 
      
  9. 
        
    9
    The ruler — map width to a percentage
    
        
    The ruler is a bar where position encodes width. Map the width range [MINW, MAXW] onto [0%, 100%]; place a tick for each breakpoint at its percentage and a marker at the current width. Clicking the bar inverts the map to set the width from the click position.
    
        
    const RANGE = MAXW - MINW;
const pct = w => (w - MINW) / RANGE * 100;         // width → %
function widthFromX(clientX){                       // % → width
  const r = ruler.getBoundingClientRect();
  const p = (clientX - r.left) / r.width;
  return MINW + Math.min(1, Math.max(0, p)) * RANGE;
}
ruler.addEventListener("pointerdown",
  e => setWidth(widthFromX(e.clientX)));
    
      
    10. 
      
  10. 
        
    10
    Two loops: rebuild (typing) vs refresh (resizing)
    
        
    Split the work. refresh() (cheap, no reload) updates readout + ruler + panels every time the width changes — that's why dragging is buttery. rebuild() re-writes the iframe and re-parses the CSS, and only runs when the user types, debounced so a burst of keystrokes reloads the frame once.
    
        
    function rebuild(){                    // on TYPING (debounced)
  pv.srcdoc  = buildDoc(html, css);
  userMedia  = parseMedia(css);
  refresh();
}
const debounce = (fn, ms) => { let t; return () => {
  clearTimeout(t); t = setTimeout(fn, ms); }; };
[htmlTa, cssTa].forEach(t => t.oninput = () => {
  html = htmlTa.value; css = cssTa.value; deb();   // deb = debounce(rebuild,300)
});
