🆔 UUID Generator — v4 random + v7 time-ordered, live

Real UUIDs, minted in your browser from real cryptographic randomness.

A UUID is a 128-bit identifier you can generate anywhere — on any laptop, phone or server — with a vanishingly small chance of ever colliding, and without asking a central database for the next number. This tool builds two flavours by hand: v4 (128 bits of crypto.getRandomValues randomness, with the version + variant bits stamped in) and v7 (a 48-bit millisecond timestamp on the front, then randomness) so the IDs sort in the order they were made. Generate one or a hundred, reformat them, then paste any UUID into the inspector to see its version, variant, and — for v7 — the exact moment it was created. All offline, no libraries.

Why v7 sorts by time

This generates 6 v7 UUIDs a few milliseconds apart, then sorts them as plain strings (A→Z). Because the timestamp lives at the front, string order equals creation order — the property that makes v7 database-index friendly. The same trick on v4 would scramble them.

🔨 Build it yourself

    
A UUID is just 16 bytes with a few bits pinned to fixed values, then printed as hex in the 8-4-4-4-12 shape. The browser hands you a correct v4 for free with crypto.randomUUID() — but hand-rolling it (and v7, which no built-in makes yet) shows you exactly where the version and variant bits go, and how a timestamp prefix turns random IDs into sortable ones. Everything below is vanilla JS, no dependencies.

    
    
      
  1. 
        
    1
    The one-liner: crypto.randomUUID()
    
        
    Every modern browser and Node ships a built-in that returns a fully-compliant version 4 UUID string — correct version and variant bits included. If all you need is a random ID, this is the answer; it uses the same cryptographic RNG you'd reach for anyway. The rest of this build exists to show what it's doing under the hood (and to make v7, which no built-in produces yet).
    
        
    crypto.randomUUID();
// "1e7c9a3f-2b4d-4a1e-9f8c-7d6e5f4a3b2c"
// v4, cryptographically random, ready to use.
// (needs a secure context: https or localhost)
    
      
    2. 
      
  2. 
        
    2
    16 random bytes (the raw material)
    
        
    A UUID is 128 bits = 16 bytes. To hand-roll v4, start by filling 16 bytes with cryptographic randomness via crypto.getRandomValues — never Math.random(), which is predictable and not built for uniqueness at scale. At this point you have a valid-looking blob; the next two steps just stamp two bytes so tools can recognise it as v4.
    
        
    const b = new Uint8Array(16);
crypto.getRandomValues(b);   // fills all 16 bytes
// b is now 128 random bits
    
      
    3. 
      
  3. 
        
    3
    Stamp the version nibble (byte 6)
    
        
    The version lives in the high 4 bits of byte 6 — that's the digit right after the second hyphen. Clear those 4 bits with & 0x0f, then set them to 4 with | 0x40. For v7 you'd use | 0x70 instead. This one nibble is how any parser instantly knows which UUID scheme it's looking at.
    
        
    b[6] = (b[6] & 0x0f) | 0x40;   // version 4
// keeps the low 4 random bits,
// forces the high 4 bits to 0100 = "4"
// (v7 would be | 0x70)
    
      
    4. 
      
  4. 
        
    4
    Stamp the variant bits (byte 8)
    
        
    The variant lives in the top bits of byte 8 — the digit after the third hyphen. RFC 4122 requires the pattern 10xx, so clear the top two bits with & 0x3f and set them with | 0x80. That's why the 17th hex digit of almost every UUID you see is 8, 9, a or b. Both v4 and v7 use the exact same variant stamp.
    
        
    b[8] = (b[8] & 0x3f) | 0x80;   // variant 10xx
// forces first two bits to "10"
// → digit becomes 8, 9, a, or b
    
      
    5. 
      
  5. 
        
    5
    16 bytes → the canonical string
    
        
    Print each byte as two lowercase hex digits, concatenate all 32, then drop hyphens into the fixed 8-4-4-4-12 positions. The hyphens are purely cosmetic — they carry no data — but the layout is standard, so keep it. This same function serialises both v4 and v7; only the bytes going in differ.
    
        
    function toStr(b){
  const h = [...b].map(x => x.toString(16).padStart(2,"0")).join("");
  return h.slice(0,8)+"-"+h.slice(8,12)+"-"+h.slice(12,16)
       +"-"+h.slice(16,20)+"-"+h.slice(20);
}
// "1e7c9a3f-2b4d-4a1e-9f8c-7d6e5f4a3b2c"
    
      
    6. 
      
  6. 
        
    6
    v7: put a 48-bit timestamp on the front
    
        
    v7's superpower is that its first 48 bits are the Unix time in milliseconds, big-endian. Take Date.now() and peel off six bytes from the low end, writing them into bytes 0–5. Because JS bitwise ops only handle 32 bits, use % 256 and Math.floor instead of shifts so the full 48-bit value survives. Now newer UUIDs always start with larger bytes.
    
        
    let ts = Date.now();               // ms since 1970
const b = new Uint8Array(16);
crypto.getRandomValues(b);
for (let i = 5; i >= 0; i--){      // bytes 0..5 = timestamp
  b[i] = ts % 256;
  ts = Math.floor(ts / 256);
}
    
      
    7. 
      
  7. 
        
    7
    v7: randomness + version 7 + variant
    
        
    Bytes 6–15 stay random (they came from getRandomValues), giving 74 bits of entropy per millisecond — plenty to avoid collisions even under heavy concurrent inserts. Then stamp version 7 into byte 6 and the same 10xx variant into byte 8, and serialise with the function from step 5. You now have a sortable, collision-resistant ID with no built-in required.
    
        
    b[6] = (b[6] & 0x0f) | 0x70;   // version 7
b[8] = (b[8] & 0x3f) | 0x80;   // variant 10xx
return toStr(b);
// "018f8c1e-9a20-7b3c-8d4e-1f2a3b4c5d6e"
//  ^^^^^^^^^^^^ = the timestamp, sortable
    
      
    8. 
      
  8. 
        
    8
    Validate + read a UUID back
    
        
    To inspect one, first check the shape with a regex, then read the version from hex digit 13 and the variant from digit 17. For a v7, the timestamp is the first 12 hex digits parsed as a base-16 number — parseInt handles 48 bits fine since it's well under JS's safe-integer limit. Feed that into new Date() and you've recovered the creation time.
    
        
    const re = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
const hex = uuid.replace(/-/g,"");
const version = parseInt(hex[12], 16);   // 4, 7, ...
if (version === 7){
  const ms = parseInt(hex.slice(0,12), 16);
  new Date(ms);   // when it was created
}
