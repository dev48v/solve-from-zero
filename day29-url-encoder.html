🔗 URL Encoder / Decoder — percent-encoding, encodeURIComponent vs encodeURI, and a query-string builder / parser

Make any text safe to put in a URL — and read it back.

A URL can only legally contain a small alphabet of characters. Everything else — spaces, accents, emoji, and the structural characters like &, ?, / and = when they appear inside a value — has to be percent-encoded: turned into its UTF-8 bytes, each byte written as % followed by two hex digits. That's why a space becomes %20 and café becomes caf%C3%A9. This page is a real, offline URL toolkit: encode a value the strict way (encodeURIComponent) or a whole link the gentle way (encodeURI) and see the difference side by side; watch a Unicode character explode into its raw %XX bytes; and build or take apart a query string one key/value pair at a time. All in the browser, no libraries.

encodeURIComponent vs encodeURI — the same input, two rules

encodeURIComponent escapes everything that isn't unreserved — including the structural characters below — so it is right for a single value or segment. encodeURI deliberately leaves those structural characters intact so a whole URL still works, so it is right for an already-built link.

Under the hood: characters → UTF-8 bytes → %XX

Unreserved characters (A–Z a–z 0–9 - _ . ~) pass through untouched. Everything else is encoded byte-by-byte: ASCII is one byte, é is two, most CJK is three, an emoji is four — one %XX per byte.

Note the note=a+b pair: in a query string a + is the form-encoded space, so the parser decodes it to a b. In a path, a + is a literal plus — same byte, different meaning depending on where it sits.

The two mistakes everybody makes

1. Encoding the whole URL with encodeURIComponent. It escapes the ://, ? and & too, so https%3A%2F%2F… is no longer a link. Encode the values, then assemble. 2. Double-encoding. Run an already-encoded string through the encoder again and %20 becomes %2520 (the % itself gets escaped to %25). Decode-then-re-encode, and decode exactly once on the way in. And always wrap decodeURIComponent in a try/catch — a stray % or %zz throws a URIError.

🔨 Build it yourself

    
A URL toolkit is three small pieces: a percent-encoder (character → UTF-8 bytes → %XX), a decoder that reverses it safely, and a pair of query helpers that build and parse ?a=1&b=2. The browser already ships encodeURIComponent / encodeURI / decodeURIComponent — use those in production — but rolling the encoder by hand once is the fastest way to see what percent-encoding is. Below is the whole thing in vanilla JS — the same code powering the LOOK tab.

    
    
      
  1. 
        
    1
    The unreserved set + a byte→%XX helper
    
        
    RFC 3986 says only these characters may appear literally with no special meaning: letters, digits, and - _ . ~. Everything else must be percent-encoded. Start with the two primitives: a test for "is this unreserved?" and a function that turns one byte into % plus two uppercase hex digits.
    
        
    const isUnreserved = ch => /[A-Za-z0-9\-_.~]/.test(ch);

// one byte -> "%" + two UPPERCASE hex digits, zero-padded
const pctByte = b => "%" + b.toString(16).toUpperCase().padStart(2, "0");

pctByte(32);   // "%20"  (space)
pctByte(233);  // "%E9"
    
      
    2. 
      
  2. 
        
    2
    Encode a character to its UTF-8 bytes
    
        
    Percent-encoding is defined over bytes, and the web's byte encoding is UTF-8. A code point below 0x80 is one byte; below 0x800 is two; below 0x10000 is three; above that (emoji, rare scripts) is four. This tiny encoder is what makes é two %XX and 😀 four.
    
        
    function utf8Bytes(str){
  const out = [];
  for (const ch of str){            // for..of walks by code point
    let c = ch.codePointAt(0);
    if (c < 0x80)       out.push(c);
    else if (c < 0x800) out.push(0xC0 | c>>6,  0x80 | c&63);
    else if (c < 0x10000)out.push(0xE0 | c>>12, 0x80 | c>>6&63, 0x80 | c&63);
    else                out.push(0xF0 | c>>18, 0x80 | c>>12&63, 0x80 | c>>6&63, 0x80 | c&63);
  }
  return out;
}
utf8Bytes("é");   // [195, 169]  -> %C3%A9
utf8Bytes("😀");  // [240,159,152,128] -> %F0%9F%98%80
    
      
    3. 
      
  3. 
        
    3
    The hand-rolled percent-encoder
    
        
    Walk the string one code point at a time. If a character is unreserved (or in an explicit safe allow-list of characters you want to keep literal), pass it through. Otherwise, expand it to UTF-8 bytes and emit a %XX for each. The safe parameter is the single knob that separates the two standard encoders.
    
        
    function percentEncode(str, safe = ""){
  let out = "";
  for (const ch of String(str)){
    if (ch.charCodeAt(0) < 128 && (isUnreserved(ch) || safe.includes(ch)))
      out += ch;                              // keep it literal
    else
      for (const b of utf8Bytes(ch)) out += pctByte(b);  // %XX per byte
  }
  return out;
}
percentEncode("a b/c");        // "a%20b%2Fc"
percentEncode("a b/c", "/");   // "a%20b/c"   (slash whitelisted)
    
      
    4. 
      
  4. 
        
    4
    The two real encoders differ only by safe
    
        
    In practice you call the built-ins — but seeing them as one function with two safe-lists demystifies them. encodeURIComponent keeps nothing structural, so it is right for a single value. encodeURI keeps the URL's delimiters so an assembled link survives. Rule of thumb: component for the parts, URI for the whole.
    
        
    // conceptually:
const encodeURIComponent = s => percentEncode(s, "!*'()");         // + unreserved
const encodeURI          = s => percentEncode(s, "!*'();,/?:@&=+$#-_.~");

encodeURIComponent("a b/c?d=1"); // "a%20b%2Fc%3Fd%3D1"   (all escaped)
encodeURI        ("a b/c?d=1");  // "a%20b/c?d=1"         (/, ?, = kept)
    
      
    5. 
      
  5. 
        
    5
    Decode — and survive malformed input
    
        
    Decoding reverses the map: find every %XX, read the byte, reassemble the UTF-8. decodeURIComponent does this — but it throws a URIError on a broken escape (%, %zz, a lone %E0). User input will be broken, so always guard it and report the failure instead of crashing.
    
        
    function safeDecode(s){
  try { return { ok: true,  value: decodeURIComponent(s) }; }
  catch (e) { return { ok: false, error: e.message }; }   // URIError
}
safeDecode("caf%C3%A9");  // { ok:true, value:"café" }
safeDecode("bad%");       // { ok:false, error:"URI malformed" }
    
      
    6. 
      
  6. 
        
    6
    Build a query string safely
    
        
    The golden rule: encode each key and each value separately, then join with the structural characters yourself. Never encode the finished string — that would escape the very = and & that give it structure. Skip empty keys so you never emit a stray =value.
    
        
    function buildQuery(pairs){          // pairs: [ [key, value], ... ]
  return pairs
    .filter(([k]) => k !== "")
    .map(([k, v]) => encodeURIComponent(k) + "=" + encodeURIComponent(v))
    .join("&");
}
buildQuery([["q","hello world"], ["tag","café"]]);
// "q=hello%20world&tag=caf%C3%A9"
    
      
    7. 
      
  7. 
        
    7
    Parse a query string back
    
        
    Reverse it: drop everything up to and including ?, cut off any # fragment, split on &, and split each pair on the first = (values can legally contain an encoded one). Decode both halves — and in a query string, turn + into a space first, the form-encoding convention.
    
        
    const dec = s => { try { return decodeURIComponent(s.replace(/\+/g, " ")); }
                   catch { return s; } };

function parseQuery(url){
  let qs = String(url).split("#")[0];          // drop fragment
  const q = qs.indexOf("?"); if (q !== -1) qs = qs.slice(q + 1);
  if (!qs) return [];
  return qs.split("&").filter(Boolean).map(pair => {
    const i = pair.indexOf("=");
    return i === -1 ? [dec(pair), ""]
                    : [dec(pair.slice(0, i)), dec(pair.slice(i + 1))];
  });
}
parseQuery("?q=hello%20world&tag=caf%C3%A9");
// [["q","hello world"], ["tag","café"]]
    
      
    8. 
      
  8. 
        
    8
    The whole toolkit + the traps to avoid
    
        
    Round-tripping proves the pieces fit: parseQuery(buildQuery(x)) gives x back. Two traps to burn in: double-encoding (encode(encode(x)) turns %20 into %2520 — decode first), and trusting input (an unencoded value can smuggle an extra &key= into your URL, or a \r\n into a redirect). Encode on the way out, decode once on the way in, validate redirect targets.
    
        
    // round-trip:
const pairs = [["a","1"], ["b","hello world"]];
parseQuery("?" + buildQuery(pairs));   // [["a","1"],["b","hello world"]] ✓

// double-encoding trap:
encodeURIComponent("hello%20world");   // "hello%2520world"  ✗ (already encoded)

// injection trap — encode the value, not the whole thing:
"?next=" + encodeURIComponent(userInput);  // safe
"?next=" + userInput;                       // open-redirect / param injection
