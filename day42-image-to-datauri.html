🖼️ Image → Data URI — drop an image, get a data: URI you can paste straight into CSS or HTML, with the real byte cost shown

Turn an image file into text you can paste anywhere — and see exactly what it costs.

A browser can render an image straight from a string instead of a URL: a data: URI like data:image/png;base64,iVBORw0KGgo… carries the whole file inline as base64 text. Paste it into background-image: url(…) or an <img src> and there is no separate network request — the bytes travel with your CSS or HTML. This is the classic trick for tiny icons and SVGs where a round-trip to the server would cost more than the pixels. The catch: base64 turns every 3 bytes into 4 characters, so the text is about 33% bigger than the file — great for a 400-byte icon, terrible for a 2 MB photo. Drop an image below (or pick one, or try the sample). Everything runs in your browser via FileReader — nothing is uploaded — and you'll get a live preview, the full data URI, a copy button, the real original → base64 → overhead byte math, and a ready-to-paste CSS snippet.

Five things people get wrong about data URIs

1. Base64 is encoding, not compression. It makes the payload bigger (≈+33%), never smaller — it exists so binary bytes survive in text-only places like a CSS file or a JSON field. If you want smaller, compress the image first, then encode. 2. Inlined images can't be cached on their own. A normal <img src="logo.png"> is fetched once and reused across pages from the browser cache; a data URI is re-sent inside every document that contains it, so inlining a big shared asset makes every page heavier. 3. Small wins, big loses. Under ~2–4 KB (icons, sprite bullets, tiny SVGs) inlining saves a whole request and usually wins; a hero photo turns into hundreds of KB of text that blocks CSS/HTML parsing. 4. The MIME type matters. data:image/png;base64,… tells the browser how to decode the bytes — get it wrong and the image won't render. FileReader fills it in from the file for you. 5. It's still just bytes. A data URI is not private or encrypted — anyone can copy the string and decode it straight back to the original file.

🔨 Build it yourself

    
An image-to-data-URI tool sounds like it needs an encoder — it doesn't. The browser hands you the whole thing in one call: FileReader.readAsDataURL(file) returns a finished data:<mime>;base64,<payload> string, MIME sniffed and base64 done. Everything else is string slicing and one division: split the string at the comma to separate the header from the payload, read file.size for the original bytes, take payload.length for the encoded bytes, and the overhead is just (encoded − original) / original. There is no library, no upload, no server — the file never leaves the page. Below is the whole tool, one honest step at a time.

    
    
      
  1. 
        
    1
    Get a File — from a drop or a picker
    
        
    Both a <input type="file"> and a drag-and-drop give you the same thing: a File object (a Blob with a name, type and size). Funnel both into one handler so the rest of the code never cares where the file came from. Guard against non-images early.
    
        
    const drop = document.getElementById("drop");
const input = document.getElementById("fileInput");

drop.onclick = () => input.click();
input.onchange = () => handleFile(input.files[0]);

drop.addEventListener("dragover", e => { e.preventDefault(); drop.classList.add("over"); });
drop.addEventListener("dragleave", () => drop.classList.remove("over"));
drop.addEventListener("drop", e => {
  e.preventDefault(); drop.classList.remove("over");
  handleFile(e.dataTransfer.files[0]);   // same entry point
});
    
      
    2. 
      
  2. 
        
    2
    FileReader.readAsDataURL — the one real call
    
        
    This is the whole engine. FileReader reads the file's bytes in the browser and, because we asked for readAsDataURL, hands back a complete data: URI — MIME type detected, bytes base64-encoded — on the async onload. Nothing is uploaded.
    
        
    function handleFile(file){
  if (!file || !file.type.startsWith("image/"))
    return showError("That's not an image file.");

  const reader = new FileReader();
  reader.onload  = e => render(file, e.target.result); // result = the data URI
  reader.onerror = () => showError("Could not read the file.");
  reader.readAsDataURL(file);   // async → onload fires with the string
}
    
      
    3. 
      
  3. 
        
    3
    Split the URI: header vs payload
    
        
    A data URI is data:<mediatype>[;base64],<data>. Everything before the first comma is the header (scheme + MIME + encoding flag); everything after is the base64 payload. One indexOf(",") and two slices split it — no parser needed.
    
        
    function parseDataURI(uri){
  const comma   = uri.indexOf(",");
  const header  = uri.slice(0, comma);     // "data:image/png;base64"
  const payload = uri.slice(comma + 1);    // "iVBORw0KGgoAAAANS..."
  const mime    = header.slice(5, header.indexOf(";")); // "image/png"
  return { header, payload, mime };
}
    
      
    4. 
      
  4. 
        
    4
    The byte math — original vs base64
    
        
    The original size is just file.size. The base64 payload is ASCII, so one character = one byte, and payload.length is the encoded byte count. Base64 emits 4 characters per 3 input bytes, so the theoretical size is 4 · ⌈n/3⌉ — the overhead is that ratio minus one, ≈ 33%.
    
        
    function byteStats(file, payload){
  const orig  = file.size;               // raw bytes
  const b64   = payload.length;          // 1 ASCII char = 1 byte
  const theory = 4 * Math.ceil(orig / 3);// 3 bytes -> 4 chars
  const overheadPct = (b64 - orig) / orig * 100;
  return { orig, b64, theory, overheadPct };
}
// e.g. 900 B image -> 1200 B base64 -> +33.3%

  5. 
        
    5
    Human-readable bytes
    
        
    Raw byte counts are hard to read past a few thousand. A tiny formatter turns them into B / KB / MB. Keep the raw number around too — you show both ("1.2 KB · 1,200 bytes") so the exact figure is never lost.
    
        
    function human(n){
  if (n < 1024)        return n + " B";
  if (n < 1024 * 1024) return (n / 1024).toFixed(1) + " KB";
  return (n / 1048576).toFixed(2) + " MB";
}
const commas = n => n.toLocaleString();   // 1200 -> "1,200"

    6
    Preview straight from the URI
    
        
    You don't need URL.createObjectURL for the preview — the data URI is a valid image source. Assign it to img.src and the browser decodes and paints it, proving the string is correct end to end.
    
        
    function render(file, uri){
  const { mime, payload } = parseDataURI(uri);
  document.getElementById("pvImg").src = uri;   // the URI renders itself
  out.value     = uri;                          // full string in the textarea
  cssOut.value  = `.icon {\n  background-image: url("${uri}");\n}`;
  htmlOut.value = `<img src="${uri}" alt="">`;
  showStats(byteStats(file, payload), mime);
}

  7. 
        
    7
    A verdict: should you inline this?
    
        
    The whole point of the tool is judgment, not just conversion. Encode a threshold: under a few KB it's a good candidate (saves a request); into the tens of KB it's borderline; past that, inlining bloats your CSS/HTML and kills caching. Turn the number into advice.
    
        
    function verdict(orig){
  if (orig <= 4 * 1024)  return ["good", "Great inline candidate — smaller than a round-trip."];
  if (orig <= 40 * 1024) return ["meh",  "Borderline — inline only if it's used on one page."];
  return ["bad", "Too big to inline — link it as a file so it can be cached."];
}

    8
    Copy to clipboard
    
        
    The output is only useful if you can grab it. navigator.clipboard.writeText copies the full string — data URI, CSS snippet, or <img> tag — and a quick "Copied!" flash confirms it, since a giant base64 blob gives no other visual feedback.
    
        
    function wireCopy(btn, getText){
  btn.onclick = () => {
    navigator.clipboard.writeText(getText());
    const old = btn.textContent;
    btn.textContent = "✓ Copied!";
    setTimeout(() => btn.textContent = old, 1200);
  };
}
