🔁 JSON ⇄ YAML Converter — a hand-rolled emitter and indentation parser, both ways, no library

The same data, two ways of writing it down — convert either direction, live.

JSON and YAML describe the exact same data model — objects, arrays, and four scalar types (string, number, boolean, null). They differ only in syntax: JSON marks structure with { }, [ ], quotes and commas; YAML marks it with indentation and a few punctuation cues, which is why config files (Kubernetes, GitHub Actions, Docker Compose) prefer it. This page is a real, offline, bidirectional converter with no library: a YAML emitter that walks a JSON tree and prints indented YAML, and a YAML parser that turns indentation back into structure. Paste either format on the left and watch the other appear — with the quoting rules, block scalars, and the famous gotchas all handled.

When does a string get quoted? — the ambiguity guard

A plain (unquoted) YAML scalar is safe only if it can't be mistaken for something else. Type a string and see how the emitter writes it as a value — and why it did or didn't wrap it in quotes.

The four things that bite everyone

1. The Norway problem. Unquoted no, yes, on, off are booleans in YAML 1.1 — so a country-code list with NO silently becomes false. Quote strings that look like keywords. 2. Tabs are illegal for indentation. YAML only allows spaces; a stray tab is a parse error, and it's invisible in most editors. 3. Comments are lost on a JSON round-trip. YAML has # comments; JSON has none. Convert YAML → JSON → YAML and every comment is gone — JSON simply has nowhere to keep them. 4. Anchors & aliases (&name / *name) let YAML reference repeated blocks; JSON has no equivalent, so they're flattened (copied) or dropped on conversion.

🔨 Build it yourself

    
A converter is two mirrored halves. The JSON side is free — the language ships JSON.parse and JSON.stringify. So the real work is the YAML side: an emitter that walks a parsed tree and prints indented YAML, and a parser that reads indentation back into a tree. In production reach for js-yaml (it handles the full 1.2 spec — anchors, tags, flow, multi-document); but rolling a working subset by hand is the fastest way to see that YAML and JSON are the same data wearing different clothes. Below is the whole thing in vanilla JS — the exact code powering the LOOK tab.

    
    
      
  1. 
        
    1
    Two formats, one data model
    
        
    JSON and YAML both encode the same four scalar types (string, number, boolean, null) plus objects and arrays. That is the whole reason a lossless conversion is even possible. So JSON.parse gives you a plain JS tree, and all you have to build is the code that prints that tree as YAML and reads it back. Start with the one predicate the emitter leans on: is this node a leaf scalar, or a container to recurse into?
    
        
    const isScalar = v => v === null || typeof v !== "object";

function emitScalar(v){
  if (v === null)  return "null";
  if (v === true)  return "true";
  if (v === false) return "false";
  if (typeof v === "number") return String(v);
  const s = String(v);
  return needsQuote(s) ? JSON.stringify(s) : s;   // quote only if ambiguous
}

    The ambiguity guard: when to quote a string
    
        
    This is the single most important function. In YAML an unquoted scalar is interpreted: 123 is a number, true is a boolean, null is null, and — the infamous Norway problem — no is false. So a string is only safe to leave bare if it cannot be read as anything else. Quote it when it looks like a number, bool, null, or date, when it starts with a YAML indicator character, or when it contains a ": " or " #" that would break the line.
    
        
    function needsQuote(s){
  if (s === "") return true;
  if (/^(null|~|true|false|yes|no|on|off)$/i.test(s)) return true;  // bool/null (Norway!)
  if (/^[-+]?(\d[\d_]*\.?\d*|\.\d+)([eE][-+]?\d+)?$/.test(s)) return true; // number-like
  if (/^\d{4}-\d{2}-\d{2}/.test(s)) return true;                    // date-like
  if (/^\s|\s$/.test(s)) return true;                               // edge whitespace
  if (/^[!&*?|>%@`"'#,\[\]{}]/.test(s)) return true;               // leading indicator
  if (/:(\s|$)/.test(s) || /\s#/.test(s)) return true;             // "key: v" / " # c" clash
  return false;
}
needsQuote("yes");   // true  -> written as "yes"
needsQuote("hello"); // false -> written bare

    The emitter: walk the tree, indent by depth
    
        
    YAML structure is indentation. So the emitter is a recursion that carries the current depth: a mapping prints key: value for scalars, or key: then its children two spaces deeper; a sequence prints one - item per element. Empty containers can't be shown by indentation (there's nothing to indent) so they fall back to the inline {} / [] flow form.
    
        
    function writeMapping(obj, indent, lines){
  const pad = " ".repeat(indent);
  for (const k of Object.keys(obj)){
    const v = obj[k], key = needsQuote(k) ? JSON.stringify(k) : k;
    if (isScalar(v))            lines.push(pad + key + ": " + emitScalar(v));
    else if (Array.isArray(v))  v.length ? (lines.push(pad+key+":"), writeSequence(v, indent+2, lines))
                                         : lines.push(pad + key + ": []");
    else                        Object.keys(v).length ? (lines.push(pad+key+":"), writeMapping(v, indent+2, lines))
                                         : lines.push(pad + key + ": {}");
  }
}

    Sequences — and objects inside them
    
        
    Each array element gets a - . A scalar is trivial; a nested array or object goes deeper. The pretty trick is an object element: emit its mapping normally, then hoist the first key onto the dash line so you get the idiomatic - id: 1 with the rest aligned underneath. That one string splice is the difference between readable and ugly YAML.
    
        
    function writeSequence(arr, indent, lines){
  const pad = " ".repeat(indent);
  for (const item of arr){
    if (isScalar(item))          lines.push(pad + "- " + emitScalar(item));
    else if (Array.isArray(item))(lines.push(pad + "-"), writeSequence(item, indent+2, lines));
    else {                       // object element
      const sub = [];
      writeMapping(item, indent + 2, sub);
      sub[0] = pad + "- " + sub[0].slice(indent + 2);   // first key onto the dash
      for (const l of sub) lines.push(l);
    }
  }
}
// -> "- id: 1\n  lead: true"

    Multi-line strings → block scalars |
    
        
    A string with newlines would be unreadable double-quoted with \n everywhere. YAML's answer is the literal block scalar |: write the text as indented lines and newlines are preserved verbatim. The trailing character is the chomping indicator — |- strips the final newline, | keeps exactly one, |+ keeps them all. (Its sibling > is the folded scalar, which turns single newlines into spaces.)
    
        
    function blockHeader(str){
  if (!str.endsWith("\n"))   return "|-";   // strip
  if (!str.endsWith("\n\n")) return "|";    // clip (one trailing \n)
  return "|+";                              // keep
}
// { note: "line one\nline two" }  ->
//   note: |-
//     line one
//     line two

    Reading a scalar back
    
        
    The parser's leaf step is emitScalar in reverse: a quoted token keeps its literal text; a bare token is coerced — null/~ to null, true/false to booleans, digits to a number, everything else stays a string. Because the emitter quoted anything ambiguous, this coercion is exactly what makes the round-trip lossless: "123" came back quoted, so it stays the string "123", while bare 123 becomes the number.
    
        
    function coerceScalar(s){
  if (s === "~" || s === "null") return null;
  if (/^true$/i.test(s))  return true;
  if (/^false$/i.test(s)) return false;
  if (/^[-+]?\d+$/.test(s)) return parseInt(s, 10);
  if (/^[-+]?(\d+\.?\d*|\.\d+)([eE][-+]?\d+)?$/.test(s) && /[.eE]/.test(s)) return parseFloat(s);
  return s;                                  // plain string
}

    The parser: indentation → structure
    
        
    The parser is recursive descent driven by the indent column. At a given indent it decides: does the line start with -  (a sequence) or contain a key: (a mapping)? A mapping reads every sibling at that same indent; when a value is empty, it peeks — a deeper line is a nested block, otherwise the value is null. Finding the key/value colon means skipping any colon inside quotes, then requiring a following space.
    
        
    function findColon(s){                 // the ": " that splits key from value
  let inS = false, inD = false;
  for (let k = 0; k < s.length; k++){
    const c = s[k];
    if (c === "'" && !inD) inS = !inS;
    else if (c === '"' && !inS) inD = !inD;
    else if (c === ":" && !inS && !inD && (k+1 >= s.length || s[k+1] === " ")) return k;
  }
  return -1;
}
const indentOf = l => { let n = 0; while (l[n] === " ") n++; return n; };

    The -  rewrite trick + the traps
    
        
    Parsing - id: 1 (a mapping that starts on the dash line) is the classic hard part. The neat shortcut: rewrite the dash and its space into plain spaces, so - id: 1 becomes   id: 1 sitting at a deeper column — then recurse as if it were an ordinary mapping. The continuation lines already align there. Verify the whole thing by round-tripping. And remember the losses: comments and anchors vanish through JSON, because JSON has no place to store them.
    
        
    // in parseSeq, for a "- rest" line at column `ind`:
let j = 1; while (content[j] === " ") j++;      // skip past "- "
const restCol = ind + j;
lines[i] = " ".repeat(restCol) + content.slice(j);   // "- id: 1" -> "  id: 1"
arr.push(parseNode(restCol));                        // recurse as a normal node
