🧹 SQL Formatter — a hand-rolled tokenizer + pretty-printer, quote-and-comment aware, no library

Paste a messy one-line query — get readable, indented SQL back.

A SQL formatter does not understand your query — it re-lays it out. But it can't just push the text through a few regexes: the word select inside a string literal, a table literally named `order`, or a -- comment must all survive verbatim. So the real work is a tokenizer that is aware of quotes and comments, followed by a printer that puts major clauses on their own lines, indents subqueries by parenthesis depth, and aligns AND/OR under WHERE. This page is that formatter, hand-rolled in vanilla JS with no library — uppercase or lowercase the keywords, choose the indent, pick a comma style, or minify back to a single line.

How the tokenizer sees your SQL — every token, colour-coded

Formatting is only safe once the text is split into tokens. Notice that a whole string literal is a single green token and a comment a single grey one — the formatter never looks inside them, which is exactly why the word select in a string is left untouched. Keywords are recognised and can be re-cased; identifiers and operators are moved around but never rewritten.

Why a find-and-replace formatter is a trap

1. Strings and comments are sacred. A naive replace(/select/gi, "SELECT") would corrupt WHERE note = 'please select one'. You must tokenize first so string and comment spans are opaque. 2. Reserved words make legal identifiers. Quoted names like "order" or `select` are columns, not clauses — casing or line-breaking them is a bug. 3. Parentheses mean two different things. ( after a function name is an argument list (keep it inline); ( before SELECT opens a subquery (break and indent). Only depth-tracking tells them apart. 4. Dialects disagree. # comments (MySQL), $$ bodies (Postgres), [bracketed] identifiers (SQL Server), || vs + concatenation — a production tool like sql-formatter is dialect-aware precisely because one regex can't cover them all.

🔨 Build it yourself

    
A formatter is two stages: a tokenizer that turns the raw text into a list of typed tokens — keeping every string, comment and quoted identifier verbatim — and a printer that walks those tokens and decides where the line breaks and indents go. There is no parsing of meaning; the layout is driven entirely by which token is a clause keyword and how deep the parentheses are. In production reach for sql-formatter (it is dialect-aware — MySQL, Postgres, T-SQL, BigQuery — and handles the edge cases a hand-rolled version won't); but building the working core yourself is the fastest way to see that pretty-printing is just re-spacing a token stream. Below is the whole thing in vanilla JS — the exact code powering the LOOK tab.

    
    
      
  1. 
        
    1
    The tokenizer: strings and comments first
    
        
    Everything hinges on this. Before you can move a single token you must slice the text into tokens, and the ones that must be captured whole are string literals ('...' with the '' escape), the two comment styles (-- line and /* block */), and quoted identifiers ("name", `name`). Scan character by character; when you hit an opening quote or comment marker, consume greedily to its close and emit one token. Because these spans become opaque, nothing downstream can ever peek inside them.
    
        
    if (c === "'"){                       // string literal, '' is an escaped quote
  let a = i; i++;
  while (i < n){ if (s[i] === "'"){ if (s[i+1] === "'"){ i += 2; continue; } i++; break; } i++; }
  toks.push({ type:"string", val:s.slice(a,i) }); continue;
}
if (c === "-" && s[i+1] === "-"){      // line comment to end of line
  let a = i; while (i < n && s[i] !== "\n") i++;
  toks.push({ type:"lcomment", val:s.slice(a,i) }); continue;
}

    2
    Words, numbers, operators, punctuation
    
        
    After the quote-aware cases, the rest is routine. A run of letters/digits/underscore is a word (a keyword or an identifier — decided later). Digits form a number (watch for 0x hex and exponents). Structural characters , ( ) ; . get their own token types because layout depends on them. Everything else — = < > + - * / and the multi-char operators <= >= <> != || :: -> — is an operator.
    
        
    if (/[0-9]/.test(c) || (c === "." && /[0-9]/.test(s[i+1]))){
  const m = /^0[xX][0-9a-fA-F]+/.exec(s.slice(i))
         || /^(?:\d+\.?\d*|\.\d+)(?:[eE][+-]?\d+)?/.exec(s.slice(i));
  toks.push({ type:"number", val:m[0] }); i += m[0].length; continue;
}
if (/[A-Za-z_@#$]/.test(c)){          // word = keyword OR identifier
  let a = i; while (i < n && /[A-Za-z0-9_@#$]/.test(s[i])) i++;
  toks.push({ type:"word", val:s.slice(a,i) }); continue;
}
// then: "," "(" ")" ";" "." get their own types; multi-char ops before single

    3
    Keyword recognition + casing
    
        
    A word is a keyword only if its uppercased form is in a known set. Tag each word with isKw/isFunc so the printer can re-case keywords (select → SELECT) while leaving identifiers verbatim. Multi-word phrases (GROUP BY, LEFT OUTER JOIN) are merged into a single logical token so they can never be split across lines.
    
        
    const canon = word.toUpperCase();
const isKw   = KEYWORDS.has(canon);
const isFunc = FUNCTIONS.has(canon);

function caseWord(tok, kwCase){
  if (tok.isKw || tok.isFunc) return kwCase === "lower" ? tok.canon.toLowerCase() : tok.canon;
  return tok.val;                     // identifiers & quoted names: never touched
}
// merge pass: ["GROUP","BY"] -> one token {canon:"GROUP BY"}

    4
    Clause-driven line breaks
    
        
    The printer keeps a current line buffer and an indent level. When it meets a major clause keyword — SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY, LIMIT, UNION — it flushes the current line and starts a fresh one at the clause indent. That single rule is what turns a one-line query into the familiar stepped shape.
    
        
    const MAJOR = new Set(["SELECT","FROM","WHERE","GROUP BY","ORDER BY",
                       "HAVING","LIMIT","OFFSET","UNION","VALUES","SET"]);
if (MAJOR.has(canon)){
  nl(frame.base);              // flush current line, indent to clause base
  put(caseWord(tok, kwCase), false);
  frame.clause = canon;
}
// nl(level) = flush(); curIndent = level;

  5. 
        
    5
    Subqueries: indent by parenthesis depth
    
        
    A ( is ambiguous. Peek at the next real token: if it is SELECT (or WITH) the paren opens a subquery, so push a new frame one level deeper and drop to a new line; otherwise it is a function/expression paren, kept inline. A frame stack makes nesting fall out naturally — each ) pops back to its opener's indent.
    
        
    const sub = nextSig(idx)?.canon === "SELECT" || nextSig(idx)?.canon === "WITH";
if (sub){
  put("(", true);
  frames.push({ base: curIndent + 1, clause:null, sub:true });
  nl(curIndent + 1);            // subquery body starts one level in
} else {
  put("(", glueToPrev);         // count(  -> stays inline
  frames.push({ base: curIndent, sub:false });
}
// on ")": const f = frames.pop(); if (f.sub){ nl(f.base - 1); put(")", false); }

  6. 
        
    6
    Column lists + comma style
    
        
    Inside SELECT, GROUP BY, ORDER BY each item goes on its own line. A comma at the clause's own depth triggers a break — but a comma inside a function's argument list belongs to a deeper (inline) frame, so it stays put. Flip between trailing commas (end of line) and leading commas (start of the next line, which makes diffs cleaner) with one branch.
    
        
    if (tok.type === "comma"){
  if (frame.commaBreak && commaStyle === "leading"){
    nl(frame.listIndent); put(",", false);      //  ", col" on the next line
  } else {
    put(",", false);                            //  "col,"  on this line
    if (frame.commaBreak) nl(frame.listIndent);
  }
}

  7. 
        
    7
    JOIN / ON layout & aligning AND/OR
    
        
    A JOIN (in all its flavours) starts a new line aligned with FROM; its ON drops one level in. Inside WHERE, HAVING and ON, each AND/OR starts a new line indented one step under the clause keyword, so a stack of conditions lines up. Outside those clauses (say a boolean in a CASE) AND stays inline.
    
        
    if (JOINS.has(canon)){ nl(frame.base); put(disp, false); frame.clause = "JOIN"; }
if (canon === "ON"){   nl(frame.base + 1); put(disp, false); frame.clause = "ON"; frame.clauseIndent = frame.base + 1; }
if (canon === "AND" || canon === "OR"){
  if (["WHERE","HAVING","ON"].includes(frame.clause)) nl(frame.clauseIndent + 1);
  put(disp, needSpace());
}

  8. 
        
    8
    Minify — the inverse — and the library
    
        
    Minifying is the same token stream with the line-break rules switched off: drop the comments (a -- line comment can't survive on one line anyway), then join the tokens with a single space where one is needed and none around . ( ) and before commas. The result is guaranteed to be a single line. Verify the whole formatter by round-tripping — minify(format(sql)) should equal minify(sql). And when the query might be MySQL one day and BigQuery the next, hand the job to sql-formatter.
    
        
    function minifySQL(sql){
  const toks = classify(tokenize(sql))
    .filter(t => t.type !== "lcomment" && t.type !== "bcomment");   // no comments on one line
  let out = "", prev = null;
  for (const tok of toks){ out += (need(prev, tok) ? " " : "") + text(tok); prev = tok; }
  return out.trim();                    // one line, always
}
// npm i sql-formatter   ->   format(sql, { language: "postgresql" })
